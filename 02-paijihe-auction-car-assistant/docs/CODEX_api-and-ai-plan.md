# 拍即合：API 与 AI 能力规划

## 当前 MVP 数据策略

当前版本使用 Mock 数据和本地计算逻辑跑通流程，不接入真实付费 API，也不提供真实交易建议。

这种做法适合求职作品集阶段：先证明产品链路、页面结构、数据字段和决策逻辑可以跑通，再考虑真实 API 和 AI 能力接入。

## 核心接口规划

| 接口 | 作用 | 当前状态 | 后续替换方向 |
| --- | --- | --- | --- |
| `getVehicleByVin` | 根据 VIN 查询车辆基础信息、车况标签、维保记录 | Mock | 接入 VIN 查询或车辆信息服务 |
| `getValuation` | 获取车辆估值区间、保值率、参考车源 | Mock + 本地估值 | 接入二手车估值 API |
| `getDistanceCost` | 根据所在地和地址估算跨区域成本 | Mock + 区域规则 | 接入地图距离 API |
| `getBidSuggestion` | 根据市场价、成交价和成本给出出价建议 | 本地规则 | 接入自建规则模型或 AI 辅助解释 |

## 数据流

```text
VIN
  -> 车辆基础信息
  -> 车况 / 维保 / 理赔记录
  -> 市场估值
  -> 三地成本
  -> 总落地成本
  -> 建议最高出价
  -> 用户实际成本反馈
```

## AI 能力可以放在哪里

这个项目不需要一开始就把所有环节都交给大模型。更合理的方式是让规则、API 和 AI 分工。

### 1. 信息解释

用户看到维保、理赔、估值和成本字段时，不一定知道它们意味着什么。AI 可以把结构化数据翻译成更容易理解的解释。

示例：

- “这条理赔记录对出价有什么影响？”
- “为什么这辆车的建议最高出价比市场均价低？”
- “我应该重点看哪些风险？”

### 2. 风险提示

AI 可以根据车况、理赔、成本和估值结果生成风险提示，但不能替用户做最终交易判断。

输出应更像“参拍前检查清单”，而不是“建议你买/不买”。

### 3. 出价建议解释

出价建议可以先由规则模型计算，再由 AI 解释理由。

这样做比直接让大模型算价格更稳。价格、距离、费用这类内容更适合结构化计算；AI 更适合负责解释、总结和提醒。

### 4. 用户反馈归因

用户上传实际成本后，AI 可以辅助归类：

- 哪些成本被低估了。
- 哪些字段经常缺失。
- 哪些地区的拖车或办理成本偏差更大。
- 哪些车型的估值误差更明显。

这些归因可以反过来优化成本规则和提示文案。

## 为什么不让 AI 直接决定出价

法拍车涉及高金额交易、法律规则、车辆真实状况和地区办理差异。公开 demo 阶段不应该让 AI 给出过强的交易判断。

更合适的边界是：

- 规则和 API 负责计算。
- AI 负责解释。
- 用户负责最终判断。
- 高风险问题提示用户找专业人士或人工确认。

## 后续迭代方向

1. 接入真实 VIN 和估值数据源。
2. 接入地图距离，替代当前区域估算规则。
3. 增加成本测算参数配置，如拖车单价、整备费区间、过户费规则。
4. 增加用户实际成本反馈表。
5. 建立 Badcase 表，记录估值偏差、成本漏项和用户误解。
6. 用 AI 生成参拍前风险清单和解释说明。

## 公开边界

本项目公开内容只展示产品方法和技术规划，不展示真实车辆交易数据、真实用户数据、真实 API Key 或任何可用于实际交易决策的确定性建议。
